01 · OVERVIEW

AI 工程的三次范式跃迁

从「把话说清楚」到「把信息给对」再到「让执行跑稳」——每一层的边界都在向外扩张，后者包含前者。

2022 — 2024

Prompt Engineering

塑造模型的局部概率空间

核心是语言设计：角色设定、风格约束、Few-shot 示例、输出格式引导。擅长短链路任务约束，不擅长弥补缺失知识。

2025

Context Engineering

解决信息供给问题

RAG 检索链路、文档切块与排序、历史对话摘要、工具返回裁剪、渐进式披露。核心是「在合适的时机把正确的信息送进去」。

2026

Harness Engineering

解决执行稳定性

当模型开始连续行动，谁来监督、约束和纠偏？Prompt 只是 Context 的一部分，Context 只是 Harness 的一部分。

关键转折：2025–2026 年 Agent 开始在生产环境中自主运行数小时、链式调用数十个工具、跨多个会话做出真实决策——暴露了前两个范式的结构性缺陷。

L3b

为什么拆 L3
L3a 推理编排是科学问题（"思考方式对不对"）；L3b 运行时调度是工程问题（"预算够不够、能不能容错"）。两者的成熟度通常差一个数量级——Hermes 的 L3b 是工业级，L3a 仍是线性 ReAct。
为什么拆 L5
L5a 观测必须实时，错过即丢；L5b 评估可异步，离线随时重算。它们的阻塞性、失败成本、技术栈都不同，合在一层意味着"L5 评分"永远是两个数字。

备注：另有一层 L0 接口与协作边界（HCI / 输出 schema / 人机切换 / 多平台集成）在严肃生产场景中也开始独立成层。本文为聚焦 Harness 内部架构，未纳入正图。

05 · 决策 ① · CONTEXT

上下文 = 一条 pipeline，不是一个字符串

最容易低估的设计。System prompt 不是写一遍就完事，而是每轮重新拼装：从静态骨架到动态边界，从目录游走到运行时配置。哪些静态、哪些每轮重算、12K token 预算分给谁——这是工程师必须自己回答的。

实物 · system prompt pipeline

SCAFFOLD · static
"You are an interactive agent..."
↓ DYNAMIC BOUNDARY ↓
DYN # Environment os · cwd · date
DYN # Project git status · last 5 commits
DYN # Agent instructions CLAUDE.md · AGENTS.md
DYN # Runtime config tools · permissions · hooks
total ≈ 12K tokens · 每轮重算

额外：directory walk 从 cwd 向上回溯 — ./CLAUDE.md · ./AGENTS.md · ~/.config/instructions.md 逐级叠加。

业界做法对比

厂商静态骨架动态注入
Claude Code系统级 scaffolddirectory walk + git 状态
Cursor内嵌 + Rules.cursorrules + 文件上下文
Aider极简 scaffoldrepo map（按 PageRank 选）
Clinescaffold + 模式MCP server 工具描述
Codex CLIOpenAI 模板AGENTS.md + 环境快照

关键决策点

  • 边界在哪：静态/动态分割决定缓存命中率
  • 预算分配：scaffold / project / instructions 各几 K
  • 查找规则：directory walk vs 显式 include vs 自动检索
反模式：把 "system prompt" 当字符串模板。一旦项目结构、git 状态、用户配置任何一项变化，模板就失效；正确做法是把它当函数：每轮重新执行，得到新串。

05 · 决策 ① · CONTEXT · 历史与缓存

历史不爆、cache 不破：滑窗 + 递归压缩

上一页讲了上下文 pipeline 的"段位安排"，留了一个真正关键的问题：对话历史会无限增长。这一页把 prompt cache 的真实作用范围 · 为什么不能全量带历史 · 怎么压 · 多次压怎么递归 四件事一次说清。

① Prompt cache 是跨 turn 的 prefix cache

// turn N+1 的请求 messages 序列：
[seg1] SCAFFOLD               ✓ hit
[seg2] tools schema           ✓ hit
[seg3] AGENTS.md              ✓ hit
[seg4] memory snapshot        ✓ hit
[seg5] history turn 1..N      ✓ hit  // append-only
─────── cache 命中边界 ────────
[seg6] turn N+1 新内容        ○ recompute

只要前缀字节一致就命中，不区分同 turn 内还是跨 turn。Hermes 能保 15+ turn 共享同一前缀，靠的就是 history 段 append-only、memory 段 turn 内冻结。

② 触发条件按 token，不按 turn

  • 一轮可能 200 tok，也可能 15K tok（read_file 一次塞满）
  • 触发条件：tokens > window × 0.5
  • 可能 turn 30 触发，也可能 turn 200 才触发

③ 切法：头 + 中间摘要 + 尾

# 触发后一次性硬切（不渐进）
[head]    turn 1~k      ~5K   # 保留最初目标
[summary] mid range     ~5K   # LLM 浓缩
[tail]    last m turns  ~10K  # 保最近上下文

④ 多次压缩 = 递归，不回头读全量 jsonl

FIRST · turn 30 触发
输入 = turn 1~25 完整 (~80K) → 输出 summary_v1 (~5K)
压完 prompt = [system] + [head turn 1~5] + [summary_v1] + [tail turn 26~30]。从 turn 31 起 append，cache 又稳定下来——直到下次触发。
SECOND · turn 65 触发
输入 = summary_v1 (5K) + turn 26~60 (~75K) → 输出 summary_v2
注意：不回去读 jsonl 的 turn 1~25，只用上次的 summary_v1。理由：① 全量重压 token 越来越贵；② 全量历史很快塞不进 context window；③ 真正重要的事实在多次压缩中会被反复保留。
THIRD · turn 100 触发
输入 = summary_v2 (5K) + turn 61~95 (~75K) → 输出 summary_v3
每次压缩输入恒定在 ~80K，summary 体量恒定在 ~5K。原始 jsonl 从未参与压缩流程——它只是 source of truth，按需查回。
写入端 · jsonl永远 append-only，全量留，从不删——source of truth。
读取端 · prompt只放窗口（最近 + 摘要），控制在 context 50~75% 内——前缀稳定，cache 多轮命中。
召回端 · session_search需要老细节就按需查 jsonl，结果作为 tool_result 末尾追加——不动前缀，cache 不破。
压缩的代价：那一刻 cache 命中边界会前移到 [head] 末尾（summary 是新写的，未在 cache 里）。但这是一次性代价——压完之后从下个 turn 起 append，新前缀稳定下来，cache 又能用 15+ 轮。关键是不要每轮微调历史，否则 cache 永远命中不了。

05 · 决策 ① · MEMORY ABC

三层 memory 的边界与分工：A · B · C

"memory" 这个词在 Agent 里挂了好几个东西。生产级实现至少分三层——A 是 session 工作面，B 是项目备忘录，C 是用户私货。三者作用域不同、生命周期不同，谁也不替谁干活。

层 作用域 典型载体 跨 session 跨项目 跨用户
A
working		 当前 session
含多个 turn / 多次启动		 prompt 内 history 段 · jsonl 归档 · session-level summary ✗ ✗ ✗
B
project		 整个项目域
团队 git 共享		 ./.hermes/MEMORY.md · AGENTS.md · .cursorrules ✓ ✗ △ 团队
C
user		 用户的所有项目
本地私货，不进 git		 ~/.config/agent/USER.md · Mem0 / Letta 云端 ✓ ✓ ✗

工业级做法：压缩 + 升级双轨——summary 压"过程"，B 记"结论"

只压不升级（脆弱）

# turn 30 触发压缩
jsonl:    1~30 全量保留
prompt:   [head] + [summary_v1] + [tail]
MEMORY.md: 没动

# 后果：
# · 重要事实压进 summary_v1
# · 下次再压会进 summary_v2
# · 多轮压缩后糊化、可能丢失

压缩 + 升级（工业级）

# turn 30 触发压缩 + nudge
jsonl:    1~30 全量保留
prompt:   [head] + [summary_v1] + [tail]
MEMORY.md: + "checkout-svc 的 tax_calc.go
             容易在边界订单出 NaN"

# 后果：
# · 这条事实从 A 升级到 B
# · 不再依赖 summary 是否记住
# · 即使 summary 糊化到 v5，结论仍清晰
A · 过程层"我刚 grep 到 ImportError" / "上次的工具调用参数"——会被压成 summary，多次压缩后糊化是正常的。需要原文走 session_search 查 jsonl。
B · 结论层"这个项目用 PostgreSQL" / "tax_calc 容易出 bug" / "DCDC 阈值是 3A"——绑项目，进 git。nudge 触发时主动从 A 升级到 B。
C · 偏好层"我习惯 ruff 不用 black" / "我看代码喜欢简洁解释"——绑你这个人，跨所有项目。clone 别人的项目不会带过来。
关键单向规则：B 和 C 不回流到 A
信息升级是单向的——A → B 或 A → C。B / C 不会反过来污染 A 的某条历史，它们只在每个 turn 启动时被读出来，作为 memory_snapshot 段参与构建 prompt（即上一页的 [seg4]）。这条单向箭头是 memory 系统稳定的关键——一旦双向，整个体系就乱。
一句话总结：A 是 session 工作面，会变、会压、用完即丢；B 是项目备忘录，跟项目走、团队共享；C 是个人私货，跟你走、跨所有项目。summary 解决"token 不爆"，B 解决"重要事实不糊化"——只压不升级，多轮后必丢；只升级不压，token 还是会爆。两轨并存，才是工业级。

05 · 决策 ③ · STATE

状态持久化：session 能不能续上

长任务、跨设备、协作场景下的真问题——agent 进程被杀以后，能否从精确的那一行恢复？2026 年的趋势是 session 与 harness 解耦，让会话成为一等公民。

实物 · JSONL append-only

.claude/projects/harness/session.jsonlAPPEND-ONLY
001 {"role": "user", "msg": "edit main.py..."}
002 {"role": "assistant", "thinking": "..."}
003 {"tool_call": "read_file", "args": {...}}
004 {"tool_result": "def add(a, b): ..."}
005 {"tool_call": "edit_file", "args": {...}}
006 {"tool_result": "ok · 1 file changed"}
007 {"compaction": {"older_turns": 2, ...}}
008 ⚠ session interrupted · process killed
009 {"resume": "session.jsonl", "from_line": 8}
010 {"tool_result": "5 passed in 0.42s"}
011 {"role": "assistant", "msg": "tests pass."}

一行一事件 · 每写 flush+sync · resume 从崩溃前最后一行开始，语义完全等价。

三种持久化方案对比

方案载体权衡
进程内内存快 / 但 crash 即丢
本地 JSONL文件 append简单、可 diff、跨进程
外部解耦独立 service多设备 / 多人 / 高可用

关键决策点

  • 恢复语义：exact replay vs 摘要重建
  • 压缩时机：何时把旧 turn 折叠为 summary
  • 访问主体：单设备 / 跨设备 / 多人协作

危险动作：三层防线组合拳

权限不是单点检查，而是静态规则 + 动态钩子 + 人在回路三层叠加。任何一层单独用都救不了——rm -rf / 在静态规则里看是 bash，需要 hook 解析参数；hook 不确定时再升级到 human approval。

┌─ WORKSPACE-WRITE ─────┐
┌─ READ-ONLY ───────┐

└───────────────────┘
+ edit_file · write_file
└───────────────────────┘
+ bash:任意 · 网络
└───────────────────────────┘

05 · 决策 ⑤ · MEMORY · 4 个子决策

记忆系统：决策 ⑤ 也由 4 个子决策构成

记忆是五个工程决策里最复杂的一个，单独展开成一章。它本身又拆成 4 个独立子决策——每个都可以独立评估、独立优化，也都可以独立出错。把会话内的 STATE 反思成跨会话的 MEMORY，正是这一章要回答的工程问题。

DECISION 1 · WRITE WHAT

决定「什么值得记」

该记：用户偏好、不可重做的副作用、错误纠正、项目级约定。
不该记：可重做的中间计算、原始对话、临时调试输出。

反模式：把每轮对话原样入库 → 信噪比拉垮，召回质量反而下降。

DECISION 2 · WRITE WHEN

决定「何时写」

推荐：周期性 nudge（每 N 轮）、压缩前 flush、用户显式 /remember、任务完成后。
避免：turn 内热写——会破坏同一轮内的前缀缓存。

Hermes 模式：写路径与读路径解耦，写不立即影响读。

DECISION 3 · READ HOW

决定「怎么进 prompt」

被动注入：稳定不变的偏好、项目事实，启动时冻结快照。
主动召回：长尾、情景性、按需查询——记忆作为 Tool 暴露（session_search）。

双轨是关键：全被动会爆炸 prompt，全主动会遗漏隐性需求。

DECISION 4 · FORGET WHEN

决定「何时忘」

必须做：冲突检测与合并、频次/时间衰减、同主题上限、不准确记忆可撤回。
后果：不遗忘 = 慢性中毒，矛盾记忆累积导致回答漂移。

Letta sleep-time compute：闲时在后台做合并 / 归纳 / 反思。

一句话原则：记忆系统的稳定性不取决于「记得多准」，而取决于「忘得多干净」。四个决策中最容易被忽视的是 #4，也是生产环境里崩最多的那个。

05 · 决策 ⑤ · MEMORY · 落地清单

5 条最佳实践与 5 个反模式

提炼自 Hermes / Mem0 / Letta / LangGraph / Anthropic / Cursor 公开生产实践——不是原则宣言，是「踩过坑」后的允诺与禁令。

DO · 最佳实践

5 条产出于生产现场的允诺

  • 写读路径解耦——写入不立即改变 prompt；读可以被动注入或者作为工具。《Hermes 的核心设计哲学》
  • 记忆快照冻结——turn 内不改，保住前缀缓存命中率。可省 cache_savings ≈ 75%。
  • 记忆即工具（主动召回）——透明、可调试，用完即弃；与被动注入双轨互补。
  • 人类可读 + 可 diff 优先——Markdown 文件（MEMORY.md / .skills/）作为语义与程序记忆底层，向量库补长尾。
  • 持续评测——跨 LoCoMo / 自建 benchmark 跨 语义·情景·多跳 三维 LLM-judge；记忆质量入指标体系。
DON’T · 反模式

5 个重复出现的败果

  • 原文入库——把每条对话全入向量库。信噪比爆炸，后续怎么调检索都救不回。
  • turn 内热写——调一次工具写一条；前缀缓存命中率归零，成本上升 4–7×。
  • 永不遗忘——矛盾与过时记忆均衡叠加，Agent「漂移」于不同时期的自己。
  • 全 vec embedding——偏好查询本该一行文件读出来，却包装成距离度圣验。慢 + 不准。
  • 记忆不过 trust boundary——外部内容未隔离即入记忆；造成 prompt injection 形成永久后门。
2026 共识：记忆是 Harness 的一个子系统，不是一个外挂。它的工程化程度决定 Agent 能否被认真用于生产——不同于「记忆能不能让模型看起来更聪明」。

06 · CASE · HERMES · L1 + L2

强项一：上下文 & 工具的工程细节

L1 / L2 决定了 Agent 第一次开口前的「装备水平」。Hermes 在这两层做的事情，几乎可以原样照搬到任何严肃的 Agent 项目里。

L1 上下文管理

  • 分层 Prompt Builder：System Identity → Skills Index → Memory Snapshot → User Project (.hermes.md / AGENTS.md / .cursorrules) → Tool Schemas → 对话历史。每一层都可独立缓存。
  • 渐进式披露 (Progressive Disclosure)：默认只塞「技能列表标题」，调用 read_skill 才展开详情——避免 200K context 被启动塞满。
  • 注入防御纵深：18 种 prompt injection 模式 + Unicode 隐形字符（U+200B/U+E0000 等 tag 字符）+ Base64 走私 + 多语言绕过。所有外部内容（URL fetch / file read）走同一过滤器。
  • 消息规范化：所有 Provider（Anthropic / OpenAI / OpenRouter）的消息差异收敛到 CanonicalMessage，保证切换模型时上下文不漂移。

L2 工具系统

  • 装饰器自注册：@register_tool 在 import 时把工具元数据写入全局 Registry，零配置发现，新增工具只需 import 一个文件。
  • MCP 协议作为外部入口：动态发现远程工具，schema 自动转换；与内置工具同样走 Registry，权限模型统一。
  • 并行只读批处理：标记 read_only=True 的工具自动并发执行（如多文件读取、grep + 列目录），写工具串行；通过路径作用域追踪避免 race。
  • 子代理委托：通过 subagent 工具开新会话——最多 3 路并发、深度 ≤ 2，独立 IterationBudget，结果回吐摘要；天然契合 L3 的预算切片。
  • 6 种执行环境：local / Docker / SSH / Modal / Daytona / Singularity 共享同一 Tool Schema，开发到生产无缝换底。
可借鉴模式：Registry + 装饰器是把工具系统做大的关键——不靠中心化注册表，而是 import 时分布式注册，工具数量增长时不需要改主流程任何一行。

06 · CASE · HERMES · L4 MEMORY

强项二：五层记忆架构——教科书级实现

这是 Hermes 最值得逐行学习的部分，也和本文第 5 章的记忆框架完全对应。它把「让 Agent 能记住事」拆成了五个独立、可替换、可降级的层。

Tier 1 · TRIGGER
Agent 循环触发器
每 10 轮 nudge 一次「该不该记？」；压缩前 flush，确保被淘汰的对话不丢关键事实；用户显式 /remember 立即写入。
Tier 2 · ORCHESTRATOR
MemoryManager（编排层）
不直接落盘，只负责路由：决定写到 MemoryStore（项目级）/ ExternalProvider（用户级）/ 仅会话内（短期）。提供统一 recall() 接口，下游层对 Agent 透明。
Tier 3 · LOCAL STORE
MemoryStore：MEMORY.md / USER.md
fcntl 文件锁 + 临时文件原子写（write→rename），保证多进程并发安全；纯文本 Markdown，人类可审计、可手改、可 git diff——这是 Honcho/Mem0 类黑盒服务给不了的。
Tier 4 · EXTERNAL PROVIDERS
8 个外部记忆插件（Mem0 / Honcho / Letta / Zep …）
Provider 容错隔离：任一外部服务故障不阻塞 Agent；统一 save() / retrieve() 接口，允许同时启用多个进行 A/B；返回结果带来源标记。
Tier 5 · CROSS-SESSION RECALL
session_search · SQLite FTS5
所有历史会话写入 messages_fts 全文索引，工具 session_search("query") 可被 Agent 主动调用——记忆变成一个工具，而不是默默注入的 prompt。
关键设计哲学：记忆的「读路径」（Tier 5 主动 search）和「写路径」（Tier 1 触发）解耦，写不一定立即影响读。这避免了「写一条记忆就污染所有未来 prompt」的常见失败模式。

Hermes 的几乎所有"奇怪"设计决策，都能被一句话解释：保住 Anthropic / OpenAI 的前缀缓存命中率。这是把 Agent 从「Demo」推到「能跑真实负载」的关键经济门槛。

决策 ① · 预算压力注入工具结果，不写消息体
消息体一变，整个前缀缓存失效。把"还剩几轮"塞到 tool_result 里，对模型可见，对缓存透明。
def tool_result_with_budget(result, budget): if budget.remaining < 5: result += f"\n[budget warning: {budget.remaining} left]" return result # ← 加在 tool_result 里，前缀不动
决策 ② · 记忆冻结快照，turn 内绝不改
一轮内可能并发调多个工具，若中途记忆改了，每次 build prompt 都是新前缀，cache 命中率归零。
memory_snapshot = freeze(memory_store.read_all()) # 整个 turn 不变 for tool_call in tool_calls: prompt = build(memory_snapshot, ...) # 同一前缀 → 缓存命中
决策 ③ · 压缩在固定阈值边界，不漂移
每轮"渐进式裁剪"会让前缀微变，永远 cache miss。改成只在 50% 阈值触发的硬切，让 15+ 轮共享同一缓存。
def compress_at_threshold(): if tokens > window * 0.5: # 只在硬阈值触发 return keep_head(10) + summarize_middle() + keep_tail(20)

三条原则一以贯之：① 任何会变的状态都不进 system prompt；② 写记忆时先 freeze 再 build，turn 内绝不改快照；③ 只在固定边界压缩（不触发就不动），让 cache 稳定 15+ 轮。

04 / 11

决策 ① 上下文主权 —— 谁在写 prompt？

Harness 的第一根支柱是主动管理 context window；SDK 用 3 个钩子把这件事产品化。

Harness 关注点

系统提示词的注入姿势

不能只给 LLM 一段裸 prompt：要分身份/工具说明/安全/风格分层组装；要能精准外科手术，也要保留护栏。

→
SDK API

system_message= 三种模式

mode=\"append\" 保留 SDK 全部护栏，仅追加角色/风格
mode=\"replace\" ⚠️ 完全替换，护栏全失效
mode=\"customize\" 按 section 改：identity / tone / safety / guidelines / …（10 段）

Harness 关注点

Workspace 上下文 + 自动事实段

CWD、git root、目录列表、可用工具清单这些 \"环境注入\"，每次 turn 都要自动拼。

→
SDK API

工作目录与配置目录

working_directory= 决定 environment_context 段
session.workspace_path 拿到归一化的运行目录
config_dir= 切换 ~/.copilot 默认配置/状态目录

Harness 关注点

压缩边界 = 缓存边界

历史无限增长时必须压缩，但压缩点要选在前缀缓存命中之后，否则 token 成本爆炸。

→
SDK API

infinite_sessions= 阈值控制

enabled=True 默认开
background_compaction_threshold=0.80 后台预压
buffer_exhaustion_threshold=0.95 硬切线

05 / 11

决策 ② 能力暴露 —— 让 LLM 看见什么工具，看不见什么

Harness 拒绝给 LLM shell(cmd: str) 这种万能口；SDK 提供 3 种工具来源 + 3 种裁剪手段。

Harness 关注点

三种工具来源

业务专属工具、跨进程能力、横向复用知识 —— 三类工具要走不同接入口，不能搅在一起。

→
SDK API

三个并列入口

tools=[ @define_tool ... ] 进程内 Python
mcp_servers={...} MCP 子进程 / HTTP
skill_directories=[...] 文件系统就是协议

Harness 关注点

能力裁剪 / 收口

主 agent 只该看到 \"路由\" 类工具；执行类工具交给 sub-agent。harness 必须能按角色裁剪。

→
SDK API

白名单 · 黑名单 · 覆写

available_tools=[...] 白名单（最高优先级）
excluded_tools=[...] 黑名单（白名单未设时生效）
overrides_built_in_tool=True 用自己的实现替换 bash / view / edit_file

Harness 关注点

主 / 子 agent 编排

每个子 agent 是独立 context window + 独立工具集，避免主 agent 上下文爆炸。

→
SDK API

Custom Agents + task 内建工具

custom_agents=[CustomAgentConfig(...)]
default_agent=DefaultAgentConfig(excluded_tools=[...]) 强制主 agent 委派
agent=\"name\" 启动时激活的角色
内建 task 工具 = 子 agent 调度入口

06 / 11

决策 ③ 状态持久化 —— 历史是 append-only 的真理

Harness 工程把 state 视为头等公民；SDK 用 events.jsonl + SQLite 索引把它默认产品化。

Harness 关注点

事件化 + 跨进程恢复

历史必须 append-only、可重放；进程崩溃后下次拉起要无缝接上同一个对话。

→
SDK API

events.jsonl + resume_session

~/.copilot/session-state/<uuid>/events.jsonl 真理源（append-only）
~/.copilot/session-store.db SQLite 索引（延迟写）
client.resume_session(sid, provider=..., on_permission_request=...)
disable_resume=True 静默接管，不触发 resume 事件

Harness 关注点

分支探索（A/B、Backup）

同一历史拉两条岔路并行实验；高风险动作前先 fork 备份。

→
SDK API

sessions.fork（experimental）

client.rpc.sessions.fork(SessionsForkRequest(session_id, to_event_id=None))
to_event_id 指定在哪个事件之前切断；返回新 session_id，resume_session 接上即可独立演化。

Harness 关注点

Workspace 虚拟化

Workspace 不一定在本地：可能是远端容器、S3、内存、数据库。所有 fs 操作必须走业务自己的实现。

→
SDK API

SessionFsProvider 抽象

继承 SessionFsProvider，实现 10 个 async 方法
create_session_fs_adapter(provider) 包成 handler
create_session_fs_handler=lambda s: adapter 注入 → 所有内建 fs 工具自动走你的 Provider

07 / 11

决策 ④ 危险动作 & HITL —— 写动作必须经人闸门

Harness 的硬约束：Agent 永远不直接执行 write。SDK 把所有副作用都路由到一个回调。

Harness 关注点

统一权限闸门

所有有副作用的工具调用都必须先停在闸门前，由 Policy / 人类来批准、拒绝或要求 \"再问一次\"。

→
SDK API

on_permission_request= + 8 种 Kind

on_permission_request: (req, inv) -> PermissionRequestResult
8 个 PermissionRequestKind：write / read / commands / mcp / mcp_sampling / custom_tool / memory
决策粒度：approve-once / reject / user-not-available / no-result
作用域：once / for-session / for-location（路径前缀）

Harness 关注点

Steering / 人在回路

必须能随时打断；agent 也应能反过来向用户问问题、要结构化输入、注册自定义指令。

→
SDK API

四种反向通道

session.abort() 随时打断当前 turn
on_user_input_request= 启用 ask_user 工具
on_elicitation_request= 结构化表单对话（JSON schema）
commands=[CommandDefinition(...)] 注册 /xxx 路由

Harness 关注点

整段生命周期 Hook

不止权限点，session 启动 / 工具调用前后 / LLM 调用前后都要能挂钩，做审计、改写、注入。

→
SDK API

hooks=SessionHooks(...)

6 类 Hook：on_session_start / on_session_end / on_tool_call（before/after）/ on_llm_call（before/after）—— 统一拦截点，可读可改可阻断。

08 / 11

决策 ⑤ 记忆与遗忘 —— 写、读、忘的工程化

短期记忆靠压缩，长期记忆靠文件。harness 把记忆做成可读、可写、可被 LLM 显式查询。

Harness 关注点

短期：压缩窗口

对话超长 → 用 LLM 把中段浓缩成 summary，保留 head 和 tail。压缩时机决定缓存命中率。

→
SDK API

infinite_sessions + Reasoning 调度

background_compaction_threshold=0.80 在缓存未失效前预压缩
buffer_exhaustion_threshold=0.95 兜底硬切
reasoning_effort=\"high\" 配合 reasoning 模型，压缩质量更高

Harness 关注点

长期：文件系统 = 知识协议

把跨会话事实写到文件，让下次 session 启动时由 SDK 自动注入到 prompt（不是 RAG，而是预加载）。

→
SDK API

Skills + AGENTS.md + customize

skill_directories=[...] 自动发现 SKILL.md 并注入 frontmatter
AGENTS.md 仓库根 → 自动并入 custom_instructions 段
system_message={\"mode\":\"customize\",\"sections\":{\"custom_instructions\":{...}}} 程序化追加

Harness 关注点

遗忘 / 截断 / 重写

有时候要主动忘掉一段（隐私、错误结论），harness 必须给出 surgical 操作而不是只能 \"全清\"。

→
SDK API

历史压缩与截断 RPC

history.compact_context_window 主动触发压缩
history.truncate 截断到指定 turn
配合 sessions.fork(to_event_id=...) 从 \"还干净\" 那一刻开新分支

11 / 11

Takeaway —— 三句话 带走

1. SDK = 产品化的 Harness

不要把 GHCP SDK 当 LLM client 用。它把 harness engineering 写死成了产品， 你只能在它指定的工位拧螺丝（回调 / 配置）。

2. 5 个决策都有对应 API

上下文 → system_message；能力 → tools/mcp/skills； 状态 → resume/fork/Fs；危险动作 → on_permission_request； 记忆 → infinite_sessions + skills。

3. 找不到对应 API ≠ 没产品化

大概率是你站错了抽象层。先翻 session-persistence / README，再考虑 hooks + RPC 自己接。
